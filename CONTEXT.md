# Project Context

The reasoning behind AiTrader. Read at the start of each session. Update when a significant decision or API quirk is discovered.

## Why This Project Exists

A 24/7 paper and live AI auto-trading platform. Four layers: static safety, adaptive strategy, DNN advisory, whale advisory. Paper trading is the default and primary training environment. Safety is the final authority and cannot be weakened by any intelligence layer.

## THE PRECEDENCE RULE (read this before adding any operator flag)

**`controls.json` (the operator's runtime override) WINS over `config/default_config.yaml` (the shipped default). On BOTH sides. Per key.**

Config says what the system SHIPS with. controls.json says what the operator DECIDED. Any flag the GUI can toggle must be read through this precedence in the C++ engine AND in the Python bridge/funnel, or one half acts on a value the other half never sees, silently.

- **Python:** read it through `llm_consensus/control_file.py`. One implementation: `overlay(cfg, block)` for a block that mirrors a config block, `flag(key, default)` for a top-level control key, `control_block(name)` for anything whose key names differ between the two files. Never open controls.json directly, and never resolve the control dir yourself.
- **C++:** mirror it in a `core/*_controls.hpp` reader. Seed the field from config, then let `bridge::json_get_bool` override it from controls.json. `layer_toggles.hpp`, `discovery_controls.hpp`, `adaptive_controls.hpp`, `sleeve_controls.hpp`, and `operator_controls.hpp` are the pattern.
- **An explicit `cfg_path` PINS a config and ignores the control file.** That is how the tests stay hermetic: without it a developer's local controls.json leaks into a test that thought it had set every value.
- **Unreadable means NO OVERRIDE, so config decides**, and config ships every operator flag off. That is the safe direction: a broken file must never start a spender. It is also why the WRITER must stay atomic, see below.
- **A flag is a REQUEST, never an authority over a hard rule.** `rl_enabled` honors this precedence, and the `rl_min_real_fills` gate is enforced at the READ (`rl_advisory.service.rl_gate_unmet`), so no file of any kind can activate RL under-gated.

Two failure modes produced this rule, both found live and both fixed (2026-07-17):

1. **A TORN READ.** `api_server/controls._write_controls` used `open(path, "w")`, which TRUNCATES then writes. Every reader swallows a read error and falls back to config, so a read landing inside that window did not fail loudly, it silently reported the SHIPPED default. **Measured: 88 percent of reads returned discovery OFF while the file on disk said ON.** It hit every runtime toggle on both sides at random, and hit hardest exactly when the operator was toggling. The write is now atomic (temp file in the same directory, then `os.replace`), so a reader always sees a complete file: the old state or the new one, never a partial one. **Any new writer of a control file must be atomic.**
2. **A CWD-RELATIVE PATH.** config ships `system.control_dir` as the relative `.control`, and three separate copies of the resolution (api_server, discovery, adaptive) each resolved it against their OWN process's working directory. The engine, the bridge, and the API backend are three processes that agreed only by all happening to launch from the repo root. A relative control_dir now resolves against the REPO ROOT, an absolute anchor every process agrees on. `MAL_CONTROL_DIR` still overrides, and an absolute value is honored as given.

## Key Decisions and Rationale

- Alpaca chosen as first real venue. Safest and easiest paper API.
- Coinbase replaces Binance for direct crypto market access. Binance does not operate in Canada. Alpaca crypto remains the paper execution venue.
- LLM council uses three models for reasoning diversity: gpt-5.5, claude-opus-4-8, gemini-3.1-pro-preview.
- Base-check gate uses Claude Haiku 4.5 (2026-07-08), replacing Gemini 3 Flash. It reuses the council's ANTHROPIC_API_KEY through the same Anthropic Messages client, so no new credential is needed. It screens signals before the paid council, cutting cost. Each gate call is well under a cent; this trades a 1,500-request-per-day free tier for near-zero paid usage, and concentrates spend on one dependency (Anthropic) instead of two.
- Council runs only on native strategy signals, never on a timer or per tick. Cost control.
- Native strategy layer blends momentum and mean reversion with a regime detector. Research shows the blend delivers smoother risk-adjusted returns across regimes than either alone.
- Whitelist limited to high-liquidity majors: BTC, ETH, SPY, QQQ. Mean reversion holds up best in liquid assets. Both strategy families fail on thin alts.
- Whale layer capped at 0.35 advisory weight. Never decisive.
- Adaptive tuner learns from REAL closed-trade PnL, not a simulator. `simulate_outcome` was removed from the default path (Task 3); the tuner may not nudge weights until ≥30 closed native trades have accumulated (thin-evidence guard). That ≥30 rule is a pure predicate (`learning/adapt_gate.hpp`) so it is unit-testable without a full Engine.
- The DNN factor is named `dnn_advisory`, not `dnn_rl`: it is a supervised advisory factor today. RL is now built as a SEPARATE module (`rl_advisory`, Stable-Baselines3 PPO) but is SHIPPED OFF — it trains only on real fills and activates only past the `rl_min_real_fills` gate (default 500), with no synthetic-data training path. Advisory sizing cap stays 0.5 regardless of which model serves.
- Model promotion is GATED: a real-data challenger only becomes champion on an explicit operator action, and only if it was trained on real data with ≥200 samples, beats the champion on walk-forward Sharpe, and has no worse drawdown (`registry.meets_promotion_criteria`). Walk-forward validation is chronological (expanding folds) — never a random split.
- Cost controls are split by owner: the C++ engine enforces daily budget + per-symbol cooldown + neutral-regime skip (`signal_engine/council_gate.cpp`); the Python side caps every provider response at `council_max_tokens`. Config surfaced via the `council:` block.
- Bridge binds loopback-only by default; remote bind requires an explicit `BRIDGE_ALLOW_REMOTE=1` opt-in. All log output is passed through credential masking (`account_manager/log_safety.py`), and a pre-commit hook scans staged content for credential shapes (Task 9).
- RL advisory was BUILT NOW but SHIPPED OFF (2026-07-05). We build the full `rl_advisory` PPO module — gym env, real-fill training gate, walk-forward eval, `/score/rl` bridge endpoint — while leaving `rl_enabled` false so the engine never calls it and the factor stays out of the ensemble entirely (`rl_advisory_factor_weight = 0.0`). Rationale: land the whole apparatus, reviewed and tested, so enabling it later is a config toggle past a real-fill gate — not a rushed build against live money. There is deliberately NO synthetic-data training path for RL: it either trains on ≥`rl_min_real_fills` real fills or it refuses. It is a separate factor from `dnn_advisory` and shares the same 0.5 advisory cap; it can never be a sole controller.
- Two council cost cuts run BEFORE the base-check gate and providers (2026-07-05), where the API spend is. (1) Risk pre-check: the C++ engine evaluates the cheap RiskGate preconditions READ-ONLY and, when a hard limit already blocks the trade, marks the payload so the council is skipped entirely (logged `risk_precheck`) — a doomed trade cannot be rescued by the council, so paying for it is waste. This reuses the existing gate result; it does not duplicate or modify gate logic. (2) Market-hours gate: outside US regular trading hours equities (SPY, QQQ) take NO new entry at all (`engine.equities_market_hours_only`, default true; logged `market_hours_entry`), while crypto trades 24/7 and is never gated. This was council-only until 2026-07-15 (see the entry below), when it was extended to the full equity entry path. Both cuts can only *skip* spend or refuse an entry, never widen risk.
- Strategy entry thresholds moved to config (2026-07-05). ADX floor, realized-vol floor, ATR floor, EMA periods, Bollinger period/std, RSI period/bounds, and the volume multiple now live in the `strategy:` block and are read by `signal_engine` (defaults keep the prior production values). Reason: the offline loop needs tunable thresholds to become a real training environment without recompiling, and every knob is validated at load so nothing loosens a Layer-1 limit.
- Synthetic-regime feed added to generate REAL native fills offline (2026-07-05, `feed_mode: synthetic_regimes`). The flat random-walk mock rarely crosses ADX/realized-vol, so it produced ~zero native trades — starving the real-fill tuner, `train_real`, and RL. The deterministic generator walks trending/range/volatile legs that cross the thresholds so BOTH momentum and mean-reversion enter. It also exposed that the native strategy's own conviction never reached the RiskGate (confidence/edge came only from mock advisory factors); we now wire the native signal in as the `rule_based` factor (it IS a rule-based signal), so a genuine technical setup can clear the gate on its merits. This changes NO gate logic or threshold and loosens no risk value.
- Historical replay added as the honest fill-generation path (2026-07-05, `feed_mode: replay`). Drives the loop from REAL bars already in the `bars` table, in chronological order, through the same closed-bar path; configurable date range + `replay_speed` (fast ignores wall-clock). It fails loudly (telling the operator to run the Alpaca backfill first) rather than silently replaying zero bars.
- Simulated clock (2026-07-05, `clock_mode: simulated`). Finite/synthetic runs finished before any 5-minute bar closed because time came from the wall clock; the simulated clock advances bar time internally so bars close on schedule. The real-clock path stays the default for the continuous, live-adjacent loop.

- native_conviction_feeds_gate flag (2026-07-10, engine block, default true). The native strategy signal drives the rule_based factor, so its conviction reaches the ensemble. True (default) preserves current behavior. False makes the gate confidence and edge come from the advisory factors alone, while the native setup still drives direction and sizing. Following the open regional market, Asia then London then NY, and trading each region's equities during its session is a real strategy given real non-overlapping regional exchange hours. The engine enforces THE safety rule in code: it only evaluates and trades an equity region a connected venue can reach, an equity order for a region with no capable venue is refused before any adapter and logged venue_unavailable_for_region, so no order is ever routed to an unreachable market. Today only NY (Alpaca US equities) is reachable, so only US equities trade during US hours, exactly as now. London and Asia are defined but venue_unavailable. - council_ran composition on the fast tier (2026-07-15, signal_engine::compose_gate_verdict). A fast-tier native entry deliberately skips the council, so the three LLM slots hold only neutral in-process mocks. Feeding those un-consulted mocks into the gate confidence dragged a genuine native conviction (0.7+) below the RiskGate min_confidence floor (0.65), capped as rule_based is at its floor share, so every fast-tier native entry was structurally blocked on confidence. Composition only, it never changes a RiskGate limit or the gate logic, and the 0.65 floor was not changed. Every risk_block event now carries the real confidence, threshold, tier, factor, and symbol in payload_json for diagnosis.

- The Finnhub health check classifies its own failures, because its token rides in the URL (2026-07-16). Every other integration authenticates with a header, so letting a raw exception reach the generic handler costs nothing. Finnhub carries the token as a query param, and the handler stringifies the exception into the `reason` the endpoint returns and the GUI renders, so a transport error naming the URL would publish the key. The check therefore catches its own failures and returns a fixed phrase (`one quote ok`, `bad key (HTTP 401)`, `rate limited (HTTP 429) after 2 retries`, `network unreachable (<type>)`, `HTTP <status>`), and a test pins the guarantee by raising an exception that CONTAINS the key and asserting the response does not. This is defense in depth, not a fix for a demonstrated leak: no realistic exception (HTTPError, URLError) puts the URL in its message today, and the guard makes that a property of the code rather than a property of which exception happened to fire. A 429 retries with the discovery client's OWN backoff policy (`finnhub_source.retry_after_seconds`, bounded, honors Retry-After) rather than a second copy, so a transient rate limit reads as busy, not broken. The check shares the 60/min free tier with a running discovery pass, which is exactly when a 429 is most likely and least meaningful.

- Feed mode and clock mode are runtime-switchable, and a switch never orphans an open position (2026-07-13). - Tuner floor keeps native entries flowing over a long run (2026-07-13). Config `adaptive.rule_based_weight_floor` (default 0.35) floors the native rule_based signal two ways: the adaptive tuner never nudges the raw rule_based weight below it (learning/adaptive.cpp, still under the 0.6 cap, still recorded to param_history), and the gate verdict (signal_engine::compose_gate_verdict, new rule_based_min_share) guarantees rule_based at least that NORMALIZED share. ## Strategy Rationale

- Momentum has deep academic support. Jegadeesh and Titman 1993 documented past winners beating past losers by roughly 12 percent annual excess return, replicated across decades and asset classes.
- Mean reversion works in range-bound liquid markets, fails in strong trends.
- No single strategy survives all regimes. A regime detector shifts weight between the two. This addresses the biggest documented failure mode.
- Transaction costs erode high-frequency mean reversion. Trade less often. Every backtest includes realistic fees and slippage.
- LLM trading evidence is promising but unproven. A 2026 audit of 77 studies found underreported costs and test sets. Council stays one advisory input with adjustable weight, earning trust through paper results.
- Connors RSI-2 is one of the most-tested short-term mean-reversion signals. The core rules: buy only inside a confirmed uptrend (price above a 200-period MA), only on RSI-2 below a low threshold (crypto needs a looser trigger, ~10, than equities, ~5), and exit when RSI-2 recovers above 65 to 70. The cross-back confirmation (wait for RSI-2 to tick back above the entry) cuts whipsaw about 20 percent in the research. A volatility band (ATR within 1 SD of its mean) and a volume filter improve the profit factor by skipping abnormal tape. RSI-2 works intraday and in range-bound tape; it fails in strong trends, which is why the regime gate routes it to range-bound and pullback conditions only.
- Time-series momentum works at longer horizons where mean reversion fails. A dual trend filter (price above both a medium and a long MA) lifts the long win rate versus a single MA. ADX above the trending threshold confirms a real trend. Momentum leads in the trending regime, RSI-2 leads in range-bound, and the two profiting in opposite regimes is the documented reason the blend smooths the equity curve.
- Crypto needs a wider stop than equities. Crypto sees large two-day selloffs even inside uptrends, so a tight fixed stop triggers before the snapback. Crypto uses a 2x ATR volatility stop; equities keep the existing ATR stop. Both stops stay inside the RiskGate's authority, which keeps its own stops unconditionally, so the wider native stop never weakens a Level-1 limit.
- Core-satellite hybrid rationale. The systematic quant core has a defined, tested edge and gets the bulk of capital (80 percent). The LLM-research satellite is a smaller, tactical allocation (20 percent) for longer-horizon discretionary ideas the systematic stack cannot express. The main failure mode of a discretionary satellite is BALLOON: a convincing thesis grows the sleeve until a single bad idea sinks the account. The hard cap prevents this mechanically, a research conviction can never override it. LLM research is promising but unproven, so the satellite is capped and starts disabled, and earns a wider allocation only through paper results, the same graduate-when-proven discipline used for RL. Any claim the satellite should exceed its cap is a reason to review the thesis quality, not to widen the cap.
- Why a discovery funnel, and why cheap-to-expensive. A fixed whitelist of four names can only find what is already on it, and most of the opportunity in a 150-name universe is in the names nobody is watching. The obstacle is not analysis, it is COST: running a full council over 150 instruments hourly would cost more than the account makes. The funnel resolves that by making the wide part free. Ranking on price, volume, volatility, momentum, gap, and a precomputed sentiment number costs nothing per instrument, so the universe can be wide without the bill being wide. Only the few names that earn it reach a paid model. The ordering is the whole design: each stage must be strictly cheaper than the one below it and must narrow, or the funnel is just an expensive loop. That is why config validation refuses survivors > finalists and council calls > survivors.
- Why the universe is curated and liquid-only. The universe is the outer edge of the funnel, not a screener over all listed instruments. Stage A ranks it for free, so width is cheap, but an illiquid name that survives to Stage C wastes a full council call on something the strategies cannot trade well. Both native families fail on thin books: RSI-2 needs a real snapback, momentum needs a real trend, and transaction costs eat the edge either way. Crypto and equities are handled differently because they behave differently: crypto composition shifts, so the active 50 are refreshed daily by liquidity and volume from a broader curated list, while large-cap equities and liquid ETFs do not churn week to week, so that list is edited by hand and stays put. The refresh ranks WITHIN the curated list, it never discovers a new coin on its own.
- Why the four levels drive both sleeves. Building a second brain for the satellite would mean two things to validate, two things to debug, and two places for a bug to cost money. The core and the satellite disagree about HORIZON, not about how to think. So the same council, DNN advisory, and whale layer evaluate both, and only the prompt framing and the horizon change: short-term-trade mode for the core, long-term-hold mode for the satellite. The advisory posture is identical in both, capped and non-decisive.
- Why the satellite goes to 30 percent while staying capped and OFF. The systematic core has a defined, tested edge and keeps the bulk of capital. The 80/20 to 70/30 move widens the ROOM the research sleeve may eventually occupy, it does not hand it capital: the sleeve still ships disabled, the hard cap still refuses an over-conviction position mechanically, and the balloon failure mode (a convincing thesis growing until one bad idea sinks the account) is still prevented in code rather than by judgement. A satellite repeatedly hitting its cap remains a thesis-quality problem to review, never a reason to widen the cap. Thirty percent is the ceiling it can earn, not the allocation it is owed.
- Why quality AND catalyst, never either alone. A long-term hold needs both a reason to own the business and a reason to own it NOW. Quality alone gives no timing, so the position sits dead through months of opportunity cost, which is exactly the failure mode that makes discretionary long-term books underperform. A catalyst alone (an earnings date, a sentiment spike) on a bad business is a gamble on a coin flip with a schedule. Requiring both is what makes the satellite's few, large, long-held positions defensible. The quality screen is deliberately loose because its job is to reject the unownable, not to pick winners. The council picks.
- Why whale does two jobs instead of one. The obvious objection to using whale data in both Stage A and Stage C is double-counting. It is not, because the two stages ask different questions of it. Surfacing is a SCREENING question (out of 119 names, which deserve a closer look) and its output is a rank. Evaluation is a DECISION question (given this name, what should we do) and its output is a verdict and a size. The same fact, that institutions are accumulating a name, is legitimate evidence for both: it makes the name worth examining, AND it is one input among several into what to conclude. The real risk would be whale deciding twice, and it does not: surfacing only buys a look, and the name still has to clear the Haiku gate and the full four levels, where whale is capped at 0.35 and cannot flip a verdict on its own. The alternative, keeping whale out of surfacing, would mean the funnel structurally cannot see a name whose only early signal is smart money moving in, which is exactly the kind of candidate a discovery engine exists to find.
- Why whale surfacing is deliberately moderate. A whale-dominant pre-screen would turn the funnel into a 13F tracker with extra steps. 13F is quarterly and ~45 days stale, so a whale-only ranking would surface what institutions bought last quarter, not what is tradeable now, and the native strategies need present-tense movement to have an edge. Weighting whale level with sentiment and native, below momentum and volatility, encodes that: whale is corroborating evidence that lifts a borderline name, not a thesis that overrides the tape. The default is a starting point, not a finding. It earns a change from paper results, the same discipline as every other weight here.
- THE ASYMMETRY, and why the react layer is safe to build at all (2026-07-16). Reacting to headlines is the highest-variance thing this system could do, and it is the one place where a model's mistake becomes an immediate trade. The resolution is not to gate that risk, it is to REMOVE it: a live event can make the engine more cautious directly (trim, exit, flag for review), and can never make it more aggressive directly. An aggressive read becomes a watchlist REFERRAL, and Stage A, Stage B, the four levels, and the RiskGate all still have to agree before anything is bought. So the worst a misread headline can do unattended is spend a screening slot and a fraction of a cent. There is no flag for event-driven entry because there is no code path to enable. That is enforced three times, independently, in two languages: adaptive/actions.py (DefensiveAction refuses to CONSTRUCT for a non-defensive action, and the queue writer takes only that type), discovery/watchlist.py (an adaptive add lands `referred`, not tradeable, status derived from source), and core/adaptive_actions.hpp (DefensiveKind has three enumerators, none aggressive; parse is an allowlist; the consumer never calls the entry path). Three independent refusals would all have to fail together.
- Why the adaptive interpreter fails CLOSED while the council gate fails OPEN (2026-07-16). llm_consensus/gate.py proceeds on an error, deliberately: a flaky cost gate must not silently suppress real analysis, and the price of failing open there is money. adaptive/interpret.py does the opposite: an error, a timeout, an unparseable reply, or a missing key all produce action="none" and severity 0. The price of failing open HERE is a position. Same vendor, same transport, opposite posture, because the consequence of the failure is different.
- Why the interpretation prompt OFFERS aggressive options (2026-07-16). The model is allowed to answer "open" or "increase", and the schema says so. The alternative is worse: if the only thing preventing an aggressive read were the prompt not mentioning it, safety would rest on prompt compliance, and every model update, jailbreak, or prompt-injected headline would be a safety incident. A news item is attacker-influenceable text by definition. So the model reports its honest read, the read is journalled, and the ARCHITECTURE refuses to turn it into an order. Safety lives in the type system and the router, not in the wording of a prompt a stranger can influence.
- Why a defensive exit does not consult the RiskGate (2026-07-16). The gate's job is to refuse risk-INCREASING orders. A gate that could refuse an exit would trap a position, which is the opposite of safety. So a defensive action routes through the same native exit accounting handle_bar_close and the sleeve rebalance already use, never a bypass and never a new order path. This is the pre-existing rule, not a new exception: exits have always executed natively.
- Honest expectations for the RSI-2 + momentum stack. Realistic target Sharpe is roughly 1.4 to 1.7 with drawdowns consistent with the Level-1 limits (daily-loss 3 percent, per-trade 0.5 percent). A backtest or paper result implying Sharpe above 3 is a signal of a methodology error to investigate (look-ahead, unpriced costs, survivorship, overfit thresholds), not a win. Every evaluation must charge realistic transaction costs and slippage, since higher frequency makes fees matter more. Realized PnL already subtracts a fee (0.01 percent per fill) before it reaches the tuner and the DNN training data, so both learn on net-of-cost outcomes, not gross.

At the old 400-token response cap it spent 384 tokens on reasoning and returned 2 characters (finishReason MAX_TOKENS), which parsed as an error. council_max_tokens was raised to 2048 (a ceiling, not spend: a real verdict used ~324 tokens total) so the council actually gets a parseable verdict. This is a cost cap, not a risk limit; the C++ side reads it only for a startup display line.
- A non-fatal startup model check (llm_consensus/model_check.py) runs at bridge startup when the real council is active. It lists each provider's models and warns if a configured council model is not reachable, so a wrong model string is a visible startup warning instead of a silent mid-trade 404. A word suffix like `-preview` is not treated as a date alias, so `gemini-3.1-pro` would NOT falsely match `gemini-3.1-pro-preview`; a date suffix like `claude-haiku-4-5-20251001` does count as the `claude-haiku-4-5` alias.
- Real council round-trip time MEASURED (2026-07-14, scripts/measure_council_latency.py, keystore keys, council_max_tokens 2048). Per provider, one call each: Haiku gate ~1.3s, GPT-5.5 ~4.6s, Claude Opus 4.8 ~2.9s, Gemini 3.1 Pro ~7.3s (the slow one, extended thinking). SEQUENTIAL total ~16.1s. This is the number the engine's bridge-call timeout must exceed. The council now scores the three providers CONCURRENTLY (llm_consensus/consensus.py _score_all), so a full council through the bridge returns in ~8.9s (gate + slowest provider) instead of ~16s, and a single slow provider only delays the council by its own time, not the sum.
- Bridge-call timeouts are config, not literals (2026-07-14, config council block). The no-trade stall was caused by the engine's default bridge-call timeout of 1500 ms being far below the ~16s real council round trip: the engine hung up mid-response, the bridge raised BrokenPipeError, the council returned no verdict, the RiskGate never got agreement, so nothing traded. Fix values: `engine_council_call_timeout_ms` 60000 (engine wait for /score/llm, well above the measured worst case with headroom), `engine_bridge_call_timeout_ms` 8000 (fast dnn/whale/rl/status calls), `provider_timeout_seconds` 30 (bridge per real provider, a hung provider fails alone after this), `gate_timeout_seconds` 15 (Haiku gate, fail-open on timeout). Config validation rejects an engine council timeout below a provider timeout. The bridge also catches BrokenPipeError / ConnectionResetError around the response write (python_bridge/server.py _send + do_POST), logging one line and never writing twice, so even if a timeout ever fires there is no double traceback.

- How each council provider actually returns its verdict, and how the parser extracts it (2026-07-14, captured from real calls). OpenAI gpt-5.5 returns clean JSON via response_format json_object (finish_reason stop). Claude Opus 4.8 returns clean JSON steered by the strict JSON system-prompt instruction. IMPORTANT: claude-opus-4-8 does NOT support an assistant-message prefill (HTTP 400 "the conversation must end with a user message"), so the common Anthropic "{" prefill trick is unavailable here, the request must end with the user message. Gemini 3.1 Pro preview uses response_mime_type application/json but is a thinking model, and it BOTCHES the closing brace: it sometimes appends a stray extra "}" after a valid object, and sometimes DROPS the closing "}" entirely (finishReason STOP, thoughts ~575 to 741 tokens, candidates ~60, total under 1100, well under the 2048 cap, so this is NOT truncation, the cap is fine). The old parser sliced the outermost { to the last }, so both quirks produced invalid JSON and an unparseable verdict. The robust parser (llm_consensus/http_json.extract_json_object) now decodes the FIRST complete JSON object at each "{" with json.JSONDecoder.raw_decode (ignoring a stray trailing brace or prose), prefers the object carrying verdict keys (direction/confidence/edge, so a thinking model's incidental object in the reasoning is skipped), and if nothing decodes it repairs an unterminated object from the first "{" (closes an open string, drops a dangling comma, appends the missing closing braces) before parsing. A genuinely broken body (for example truncated mid-number) still returns None and the provider flat-falls-back with source error. ## GUI Plan (dashboard/control-surface overhaul — next major effort)

The dashboard is a first-class control surface, not a read-out. The overhaul (after paper-loop stability is proven) exposes the layered decision system as direct controls, with one inviolable rule: **no control in the UI can weaken the RiskGate.** Layer-1 hard limits are editable only through the typed-`CONFIRM` L1 editor and are still validated at load (a change that would enlarge a loss/exposure ceiling, shrink a quality gate, or disable a safety toggle is rejected); every other control is subordinate.

- **Per-layer enable toggles, safety ALWAYS ON.** Each decision layer (adaptive, LLM council, `dnn_advisory`, whale, `rl_advisory`) gets an on/off toggle. The Layer-1 static-safety layer has NO off switch — it is rendered on and locked. Turning a layer off drops its factor from the ensemble; it can never turn off the gate.
- **Per-model council toggles.** Each council slot (OpenAI / Anthropic / Google) toggles independently, plus the Claude Haiku base-check gate, so the operator can run a cheaper or narrower council without code changes.
- **Champion/challenger controls with MANUAL promote + rollback.** For `dnn_advisory` (and `rl_advisory` when enabled): view champion vs challenger metrics (walk-forward Sharpe, drawdown, provenance, n_samples), a **manual Promote** button gated on `meets_promotion_criteria`, and a **Rollback** to the prior champion. Promotion is never automatic.
- **RL enable toggle, default OFF, behind the 500-fill gate.** A dedicated `rl_enabled` switch that stays off by default and is disabled in the UI until the real closed-fill count reaches `rl_min_real_fills` (default 500); the panel shows current fills / gate. No synthetic-data training path is ever offered.
- **Weight sliders grouped by layer.** Ensemble weights presented grouped by layer (rule-based, LLM, dnn_advisory, whale, rl_advisory), each with lock-against-adaptive and enable/disable, and a live "preview verdict on weight change" before applying.
- **Regime override.** Manual override of the detected regime (trend / range / neutral) for testing and for operator judgement, with a clear "auto" default and a visible banner when overridden.
- **Budget dial.** Operator-adjustable council daily budget + per-symbol cooldown (the cost controls), surfaced as a dial with the current spend against it.
