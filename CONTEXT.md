# Project Context

The reasoning behind AiTrader. Read at the start of each session. Update when a significant decision or API quirk is discovered.

## Why This Project Exists

A 24/7 paper and live AI auto-trading platform. Four layers: static safety, adaptive strategy, DNN advisory, whale advisory. Paper trading is the default and primary training environment. Safety is the final authority and cannot be weakened by any intelligence layer.

## Key Decisions and Rationale

- Alpaca chosen as first real venue. Safest and easiest paper API.
- Coinbase replaces Binance for direct crypto market access. Binance does not operate in Canada. Alpaca crypto remains the paper execution venue.
- LLM council uses three models for reasoning diversity: gpt-5.5, claude-opus-4-8, gemini-3.1-pro-preview.
- Base-check gate uses Claude Haiku 4.5 (2026-07-08), replacing Gemini 3 Flash. It reuses the council's ANTHROPIC_API_KEY through the same Anthropic Messages client, so no new credential is needed. It screens signals before the paid council, cutting cost. Each gate call is well under a cent; this trades a 1,500-request-per-day free tier for near-zero paid usage, and concentrates spend on one dependency (Anthropic) instead of two.
- Council runs only on native strategy signals, never on a timer or per tick. Cost control.
- Native strategy layer blends momentum and mean reversion with a regime detector. Research shows the blend delivers smoother risk-adjusted returns across regimes than either alone.
- Whitelist limited to high-liquidity majors: BTC, ETH, SPY, QQQ. Mean reversion holds up best in liquid assets. Both strategy families fail on thin alts.
- Whale layer capped at 0.35 advisory weight. Never decisive.
- Adaptive tuner learns from REAL closed-trade PnL, not a simulator. `simulate_outcome` was removed from the default path (Task 3); the tuner may not nudge weights until ≥30 closed native trades have accumulated (thin-evidence guard). That ≥30 rule is a pure predicate (`learning/adapt_gate.hpp`) so it is unit-testable without a full Engine.
- The DNN factor is named `dnn_advisory`, not `dnn_rl`: it is a supervised advisory factor today. RL is now built as a SEPARATE module (`rl_advisory`, Stable-Baselines3 PPO) but is SHIPPED OFF — it trains only on real fills and activates only past the `rl_min_real_fills` gate (default 500), with no synthetic-data training path. Advisory sizing cap stays 0.5 regardless of which model serves.
- Model promotion is GATED: a real-data challenger only becomes champion on an explicit operator action, and only if it was trained on real data with ≥200 samples, beats the champion on walk-forward Sharpe, and has no worse drawdown (`registry.meets_promotion_criteria`). Walk-forward validation is chronological (expanding folds) — never a random split.
- Cost controls are split by owner: the C++ engine enforces daily budget + per-symbol cooldown + neutral-regime skip (`signal_engine/council_gate.cpp`); the Python side caps every provider response at `council_max_tokens`. Config surfaced via the `council:` block.
- Bridge binds loopback-only by default; remote bind requires an explicit `BRIDGE_ALLOW_REMOTE=1` opt-in. All log output is passed through credential masking (`account_manager/log_safety.py`), and a pre-commit hook scans staged content for credential shapes (Task 9).
- RL advisory was BUILT NOW but SHIPPED OFF (2026-07-05). We build the full `rl_advisory` PPO module — gym env, real-fill training gate, walk-forward eval, `/score/rl` bridge endpoint — while leaving `rl_enabled` false so the engine never calls it and the factor stays out of the ensemble entirely (`rl_advisory_factor_weight = 0.0`). Rationale: land the whole apparatus, reviewed and tested, so enabling it later is a config toggle past a real-fill gate — not a rushed build against live money. There is deliberately NO synthetic-data training path for RL: it either trains on ≥`rl_min_real_fills` real fills or it refuses. It is a separate factor from `dnn_advisory` and shares the same 0.5 advisory cap; it can never be a sole controller.
- Two council cost cuts run BEFORE the base-check gate and providers (2026-07-05), where the API spend is. (1) Risk pre-check: the C++ engine evaluates the cheap RiskGate preconditions READ-ONLY and, when a hard limit already blocks the trade, marks the payload so the council is skipped entirely (logged `risk_precheck`) — a doomed trade cannot be rescued by the council, so paying for it is waste. This reuses the existing gate result; it does not duplicate or modify gate logic. (2) Market-hours skip: equities (SPY, QQQ) skip the gate+council outside US regular trading hours (`engine.equities_market_hours_only`, default true; logged `market_hours`), while crypto trades 24/7 and is never skipped. Both are pure cost controls — they can only *skip* spend, never widen risk.
- Strategy entry thresholds moved to config (2026-07-05). ADX floor, realized-vol floor, ATR floor, EMA periods, Bollinger period/std, RSI period/bounds, and the volume multiple now live in the `strategy:` block and are read by `signal_engine` (defaults keep the prior production values). Reason: the offline loop needs tunable thresholds to become a real training environment without recompiling, and every knob is validated at load so nothing loosens a Layer-1 limit.
- Synthetic-regime feed added to generate REAL native fills offline (2026-07-05, `feed_mode: synthetic_regimes`). The flat random-walk mock rarely crosses ADX/realized-vol, so it produced ~zero native trades — starving the real-fill tuner, `train_real`, and RL. The deterministic generator walks trending/range/volatile legs that cross the thresholds so BOTH momentum and mean-reversion enter. It also exposed that the native strategy's own conviction never reached the RiskGate (confidence/edge came only from mock advisory factors); we now wire the native signal in as the `rule_based` factor (it IS a rule-based signal), so a genuine technical setup can clear the gate on its merits. This changes NO gate logic or threshold and loosens no risk value.
- Historical replay added as the honest fill-generation path (2026-07-05, `feed_mode: replay`). Drives the loop from REAL bars already in the `bars` table, in chronological order, through the same closed-bar path; configurable date range + `replay_speed` (fast ignores wall-clock). It fails loudly (telling the operator to run the Alpaca backfill first) rather than silently replaying zero bars.
- Simulated clock (2026-07-05, `clock_mode: simulated`). Finite/synthetic runs finished before any 5-minute bar closed because time came from the wall clock; the simulated clock advances bar time internally so bars close on schedule. The real-clock path stays the default for the continuous, live-adjacent loop.
- Alpaca is a PAPER + MARKET-DATA venue only. It has no live brokerage path and must never be wired to one. The feed/replay work reads Alpaca market data and paper fills only; the simulation controls never touch execution mode.

Polymarket is off the venue lists, config, credentials, dashboard groups, and the whale sources. A guard test (tests/test_no_polymarket.py) fails if a reference returns to runtime code or config.
- Alpaca paper is the primary ONLINE loop (2026-07-06, `feed_mode: alpaca_paper`). It forces the online AlpacaFeed, so the loop evaluates real 5-minute Alpaca bars over the bridge and sends paper orders to Alpaca paper, through the same closed-bar path the offline feeds use. The other feed modes (synthetic_regimes, replay, flat_random_walk) stay offline test tools. Alpaca still has no live path.

- native_conviction_feeds_gate flag (2026-07-10, engine block, default true). The native strategy signal drives the rule_based factor, so its conviction reaches the ensemble. True (default) preserves current behavior. False makes the gate confidence and edge come from the advisory factors alone, while the native setup still drives direction and sizing. Composition only, in signal_engine::compose_gate_verdict. It never changes a RiskGate limit or the gate logic.

- Native strategy warms from a real-bar backfill before it evaluates (2026-07-13). The engine seeds its in-memory bar history from the `bars` table on construction, so a live paper run starts with warm indicators instead of a cold start. `scripts/start_paper_trading.sh` backfills the whitelist first (the Alpaca 30-day 5-min window is thousands of bars per symbol, far more than the ~102 the longest lookback needs) and verifies every symbol warm before opening the loop. `strategy::min_bars_to_warm` is the longest indicator lookback (the 100-period EMA plus 2, which dominates ADX, ATR, Bollinger, RSI, volume, and realized vol). The startup block prints a per-symbol per-indicator warm/cold line.
- Warm-state gate on the real path (2026-07-13). On feed_mode alpaca_paper the engine does not evaluate a symbol for a native entry until its indicators are warm (`Engine::symbol_is_warm`, the pure `strategy::indicators_warm` predicate). A cold symbol waits and never fires on partial data. Cold/warm transitions log to the event log as `warm_state`. Offline feed modes keep building history for tests and are not gated, so existing behavior is unchanged.
- Feed mode and clock mode are runtime-switchable, and a switch never orphans an open position (2026-07-13). The engine reads `feed_mode` and `clock_mode` from controls.json each loop iteration (core/feed_clock.hpp), the same control-file pattern as the layer toggles, and `run_forever` re-dispatches on the resulting feed each iteration (tick modes poll the feed, bar modes step a bar). A clock switch applies immediately. A feed switch AWAY from alpaca_paper while a paper position is open is BLOCKED (the safest option: the position keeps being managed by its native exits on the current feed, never stranded on a feed about to be replaced), logged as `feed_mode_blocked`; the alternative of force-flattening was rejected as heavier and more surprising. A switch INTO alpaca_paper re-arms the warm-start gate. Every applied switch logs `feed_mode` / `clock_mode` with old and new. A missing or invalid controls.json value keeps the launch feed/clock, so a missing file never forces an offline run onto the live feed. The GUI Controls page carries a validated `/controls/feed_clock` endpoint (refuses an unsafe switch, audited), and the run-state banner plus the top status strip mirror the current feed/clock. Runtime strict-mode enforcement is at startup only: a runtime switch into alpaca_paper logs the switch and re-arms the warm gate but does not re-run the strict reachability check (it never crashes a running loop).

- Tuner floor keeps native entries flowing over a long run (2026-07-13). Config `adaptive.rule_based_weight_floor` (default 0.35) floors the native rule_based signal two ways: the adaptive tuner never nudges the raw rule_based weight below it (learning/adaptive.cpp, still under the 0.6 cap, still recorded to param_history), and the gate verdict (signal_engine::compose_gate_verdict, new rule_based_min_share) guarantees rule_based at least that NORMALIZED share. The share floor is the part that actually sustains entries: a raw floor alone fails because the five advisory factors saturate at the 0.6 cap and drive rule_based's normalized share toward zero, dropping the gate confidence below the RiskGate minimum (0.65) and stalling native entries near 30 fills. With the share floor a 5000-step synthetic run keeps generating native entries past 100 closed trades. This is an advisory weight bound. It touches no risk limit and never weakens a Layer-1 gate, the RiskGate still judges every order on its own thresholds (a residual cap in the synthetic mock is the required_model_agreement_count gate, since the momentum-biased mock advisory factors disagree with mean-reversion entries, that is a mock-data artifact, not the tuner).
- The engine consumes every remaining controls.json override (2026-07-13). Model toggles, regime pins, and budget are consumed by the C++ engine each iteration (core/operator_controls.hpp, consume_operator_controls), the same control-file pattern as the layer toggles. Model toggles drop a disabled provider slot from the council for the iteration (flat keys llm_primary/secondary/tertiary_enabled derived from the models map), the ensemble math handles the reduced set, and if every provider is disabled the council falls back to a clearly logged skip (at least one provider required). Regime pins override the detector for a pinned symbol (flat key regime_pin:<symbol>, test-only, valid labels only), affecting the surfaced regime and the council neutral-skip. Budget adjusts council_daily_budget and the per-symbol cooldown at runtime within the server-side bounds (flat keys rt_council_daily_budget / rt_per_symbol_cooldown_minutes, re-clamped defensively). Each change logs old and new. Safe decision documented: promote and rollback of the dnn champion execute in Python at the audited endpoint (api_server/controls.py request_promote/request_rollback through ml_factor.registry.promote / rollback), not polled by the C++ engine each iteration, because the model_registry and meets_promotion_criteria are Python-owned and the engine cannot re-check the criteria without duplicating that logic. Promotion stays gated by meets_promotion_criteria (a runtime promote cannot bypass it) and the RL fill gate is unchanged.
- native_conviction_feeds_gate recommendation for the paper week (2026-07-13). Keep it at the default TRUE for the paper week. True feeds the native rule_based conviction into the gate confidence and edge, which is what the new rule_based share floor relies on to keep native entries clearing the gate over a long run, so fills keep accumulating for training. The tradeoff is a mild double-count: the native setup drives BOTH direction and sizing AND the gate confidence and edge. Set it false only if a specific reason favors taking the gate confidence and edge from the advisory factors alone (for example, once a trained real council is trusted more than the native technicals), noting that with the flag false the rule_based share floor is moot because rule_based is excluded from the gate confidence and edge. The flag stays operator-controlled and was not changed this session.
- Simulated-clock market-hours skip and replay cooldown now use the right time source (2026-07-13). The equities market-hours council skip keys off the simulated timestamp under clock_mode simulated and real wall-clock under clock_mode real (core/engine.cpp passes now_epoch when simulated_clock_ is set, else std::time). Replay council-cooldown spacing keys off the true historical bar ts (util::iso8601_to_epoch parses the stored timestamp), matching how the per-day trade cap already works, rather than a synthetic sequential epoch.

- Paper trading starts and stops from the GUI through a supervisor (2026-07-14). A small backend supervisor (api_server/supervisor.py) owns the bridge + engine lifecycle and exposes GET /engine/state, POST /engine/start, POST /engine/stop on the existing loopback backend. It runs the SAME sequence as scripts/start_paper_trading.sh (backfill, warm-verify, bridge, engine feed_mode alpaca_paper clock real, health checked between steps) through one shared callable, api_server/stack.py, so the script and the GUI never drift. The script was refactored to reuse stack (warm-report CLI, seed_feed_clock, the lock) instead of its inline copies. State reports not_running, starting, warming (with per-symbol warm progress), running, stopping. Start takes a confirm step and is disabled while already running or starting. Strict no-silent-mock still holds: an on-real layer that is unreachable makes the engine exit on start, and the supervisor surfaces that failure with what is missing rather than a silent mock. Single-instance guard: a shared .control/engine.lock records the engine and bridge pids (written by BOTH the script and the supervisor), a start is refused when a live lock exists (script or a prior GUI start), and a stale lock from a crashed run is detected by a dead pid and cleared on the next start. GUI Stop is a GRACEFUL shutdown of the processes the supervisor started. It is a different mechanism from the safety halt.
- The kill switch stays independent of the GUI and the supervisor (2026-07-14). The C++ engine reads the kill-request control file itself at the top of every loop iteration, so a kill halts the engine even with the GUI, the backend, and the supervisor all down. A safety halt never depends on either. The supervisor and its stack module never read or write the kill-request file (asserted in tests), so start/stop can never interfere with the safety halt, and the always-visible kill switch is never replaced by the Stop button. The lock file lives next to the kill-request file in the control dir but is a distinct file the kill path never touches.

- GUI supervisor start survives past warming via bridge env parity and readiness gating (2026-07-14). The GUI start died right after warming because the supervisor spawned the bridge WITHOUT the whale env flags the script exports. The whale library treats SEC_EDGAR_ENABLED and WHALE_LIVE_ENABLED as env opt-ins that default OFF, so without them the bridge reported whale_real=false, the engine's strict on-real whale check refused, the engine exited, and the supervisor tore the bridge down too. Fix: the supervisor now spawns the bridge with stack.bridge_env() (the port plus the whale flags from config), the SAME helper the start script now uses via the stack bridge-env-export CLI, so the two cannot drift. The supervisor also gates the engine start on bridge readiness: it waits for the bridge /health to pass (60s window, a slow health check is not a failure), then confirms via stack.bridge_missing_real_layers that the on-real layers the engine will require are actually served real, and fails HERE with a specific reason (surfaced to the GUI state.error and the event log as engine_supervisor) instead of letting the engine exit cryptically. A teardown only fires on a real unrecoverable error, never on a normal warming transition. Verified live from a cold GUI: starting to warming to running in ~18s, still running after 25s, bridge council_real=dnn_real=whale_real=sec_edgar=True, graceful stop returns not_running. run_gui.sh documents that the backend hosts the supervisor and must run first, and the Start button drives it (it never launches the backend it runs inside).

- The start script and the GUI supervisor self-clean stale processes and ports pre-flight, and track PIDs for clean teardown (2026-07-14). ## Success Criteria (pre-registered, week of 2026-07-15)

Written before the week starts, so the result is judged against a fixed bar, not a moved one.

- Research sleeve: judged on THESIS QUALITY and position behavior, not on the same scalp metric as the quant core. If the reasoning is thin or the long-term holds behave poorly, the sleeve stays capped or gets cut. It earns a wider allocation only through demonstrated paper results.
- Discipline: mid-week changes are limited to genuine trading-blocking DEFECTS only. Everything else (tuning, feature ideas, cadence tweaks) gets logged and applied AFTER the week, so the week is a clean read.

## Strategy Rationale

- Momentum has deep academic support. Jegadeesh and Titman 1993 documented past winners beating past losers by roughly 12 percent annual excess return, replicated across decades and asset classes.
- Mean reversion works in range-bound liquid markets, fails in strong trends.
- No single strategy survives all regimes. A regime detector shifts weight between the two. This addresses the biggest documented failure mode.
- Transaction costs erode high-frequency mean reversion. Trade less often. Every backtest includes realistic fees and slippage.
- LLM trading evidence is promising but unproven. A 2026 audit of 77 studies found underreported costs and test sets. Council stays one advisory input with adjustable weight, earning trust through paper results.
- Connors RSI-2 is one of the most-tested short-term mean-reversion signals. The core rules: buy only inside a confirmed uptrend (price above a 200-period MA), only on RSI-2 below a low threshold (crypto needs a looser trigger, ~10, than equities, ~5), and exit when RSI-2 recovers above 65 to 70. The cross-back confirmation (wait for RSI-2 to tick back above the entry) cuts whipsaw about 20 percent in the research. A volatility band (ATR within 1 SD of its mean) and a volume filter improve the profit factor by skipping abnormal tape. RSI-2 works intraday and in range-bound tape; it fails in strong trends, which is why the regime gate routes it to range-bound and pullback conditions only.
- Time-series momentum works at longer horizons where mean reversion fails. A dual trend filter (price above both a medium and a long MA) lifts the long win rate versus a single MA. ADX above the trending threshold confirms a real trend. Momentum leads in the trending regime, RSI-2 leads in range-bound, and the two profiting in opposite regimes is the documented reason the blend smooths the equity curve.
- Crypto needs a wider stop than equities. Crypto sees large two-day selloffs even inside uptrends, so a tight fixed stop triggers before the snapback. Crypto uses a 2x ATR volatility stop; equities keep the existing ATR stop. Both stops stay inside the RiskGate's authority, which keeps its own stops unconditionally, so the wider native stop never weakens a Level-1 limit.
- Core-satellite hybrid rationale. The systematic quant core has a defined, tested edge and gets the bulk of capital (80 percent). The LLM-research satellite is a smaller, tactical allocation (20 percent) for longer-horizon discretionary ideas the systematic stack cannot express. The main failure mode of a discretionary satellite is BALLOON: a convincing thesis grows the sleeve until a single bad idea sinks the account. The hard cap prevents this mechanically, a research conviction can never override it. LLM research is promising but unproven, so the satellite is capped and starts disabled, and earns a wider allocation only through paper results, the same graduate-when-proven discipline used for RL. Any claim the satellite should exceed its cap is a reason to review the thesis quality, not to widen the cap.
- Honest expectations for the RSI-2 + momentum stack. Realistic target Sharpe is roughly 1.4 to 1.7 with drawdowns consistent with the Level-1 limits (daily-loss 3 percent, per-trade 0.5 percent). A backtest or paper result implying Sharpe above 3 is a signal of a methodology error to investigate (look-ahead, unpriced costs, survivorship, overfit thresholds), not a win. Every evaluation must charge realistic transaction costs and slippage, since higher frequency makes fees matter more. Realized PnL already subtracts a fee (0.01 percent per fill) before it reaches the tuner and the DNN training data, so both learn on net-of-cost outcomes, not gross.

## API Notes and Quirks

- Council model reachability verified 2026-07-12 (scripts/list_provider_models.sh, keystore-first keys). Reachable ids in use: OpenAI `gpt-5.5` (in the account's model list), Anthropic `claude-opus-4-8` (in the list) and `claude-haiku-4-5` (server-resolved alias of `claude-haiku-4-5-20251001`), Google `gemini-3.1-pro-preview`. Substitution: the previously configured `gemini-3.1-pro` does NOT exist for the key (HTTP 404, "not found for API version v1beta"); the reachable Gemini 3.1 Pro id is `gemini-3.1-pro-preview`, so the tertiary slot uses that. OpenAI needed no model change, only a request-shape fix.
- OpenAI GPT-5 family request shape: the earlier HTTP 400 was not auth and not the model. Two shape rules: (1) the token cap field is `max_completion_tokens`, not the deprecated `max_tokens` ("Unsupported parameter: 'max_tokens' is not supported with this model"); (2) only the default temperature is allowed ("'temperature' does not support 0.2 with this model. Only the default (1) value is supported"), so the provider omits temperature. The provider keeps `response_format: json_object`, and the system prompt contains the word "JSON" as that mode requires.
- Thinking models need output headroom. Gemini 3.1 Pro is thinking-only (`thinkingBudget: 0` is rejected). At the old 400-token response cap it spent 384 tokens on reasoning and returned 2 characters (finishReason MAX_TOKENS), which parsed as an error. council_max_tokens was raised to 2048 (a ceiling, not spend: a real verdict used ~324 tokens total) so the council actually gets a parseable verdict. This is a cost cap, not a risk limit; the C++ side reads it only for a startup display line.
- A non-fatal startup model check (llm_consensus/model_check.py) runs at bridge startup when the real council is active. It lists each provider's models and warns if a configured council model is not reachable, so a wrong model string is a visible startup warning instead of a silent mid-trade 404. A word suffix like `-preview` is not treated as a date alias, so `gemini-3.1-pro` would NOT falsely match `gemini-3.1-pro-preview`; a date suffix like `claude-haiku-4-5-20251001` does count as the `claude-haiku-4-5` alias.
- Real council round-trip time MEASURED (2026-07-14, scripts/measure_council_latency.py, keystore keys, council_max_tokens 2048). Per provider, one call each: Haiku gate ~1.3s, GPT-5.5 ~4.6s, Claude Opus 4.8 ~2.9s, Gemini 3.1 Pro ~7.3s (the slow one, extended thinking). SEQUENTIAL total ~16.1s. This is the number the engine's bridge-call timeout must exceed. The council now scores the three providers CONCURRENTLY (llm_consensus/consensus.py _score_all), so a full council through the bridge returns in ~8.9s (gate + slowest provider) instead of ~16s, and a single slow provider only delays the council by its own time, not the sum.
- Bridge-call timeouts are config, not literals (2026-07-14, config council block). The no-trade stall was caused by the engine's default bridge-call timeout of 1500 ms being far below the ~16s real council round trip: the engine hung up mid-response, the bridge raised BrokenPipeError, the council returned no verdict, the RiskGate never got agreement, so nothing traded. Fix values: `engine_council_call_timeout_ms` 60000 (engine wait for /score/llm, well above the measured worst case with headroom), `engine_bridge_call_timeout_ms` 8000 (fast dnn/whale/rl/status calls), `provider_timeout_seconds` 30 (bridge per real provider, a hung provider fails alone after this), `gate_timeout_seconds` 15 (Haiku gate, fail-open on timeout). Config validation rejects an engine council timeout below a provider timeout. The bridge also catches BrokenPipeError / ConnectionResetError around the response write (python_bridge/server.py _send + do_POST), logging one line and never writing twice, so even if a timeout ever fires there is no double traceback.

- How each council provider actually returns its verdict, and how the parser extracts it (2026-07-14, captured from real calls). OpenAI gpt-5.5 returns clean JSON via response_format json_object (finish_reason stop). Claude Opus 4.8 returns clean JSON steered by the strict JSON system-prompt instruction. IMPORTANT: claude-opus-4-8 does NOT support an assistant-message prefill (HTTP 400 "the conversation must end with a user message"), so the common Anthropic "{" prefill trick is unavailable here, the request must end with the user message. Gemini 3.1 Pro preview uses response_mime_type application/json but is a thinking model, and it BOTCHES the closing brace: it sometimes appends a stray extra "}" after a valid object, and sometimes DROPS the closing "}" entirely (finishReason STOP, thoughts ~575 to 741 tokens, candidates ~60, total under 1100, well under the 2048 cap, so this is NOT truncation, the cap is fine). The old parser sliced the outermost { to the last }, so both quirks produced invalid JSON and an unparseable verdict. The robust parser (llm_consensus/http_json.extract_json_object) now decodes the FIRST complete JSON object at each "{" with json.JSONDecoder.raw_decode (ignoring a stray trailing brace or prose), prefers the object carrying verdict keys (direction/confidence/edge, so a thinking model's incidental object in the reasoning is skipped), and if nothing decodes it repairs an unterminated object from the first "{" (closes an open string, drops a dangling comma, appends the missing closing braces) before parsing. A genuinely broken body (for example truncated mid-number) still returns None and the provider flat-falls-back with source error. Measured parse success: originally the Gemini stray/missing-brace shapes were unparseable, after the fix all three providers return parseable real verdicts 12/12 across a 4-calls-each run and a full council through the bridge returns agreement 3 with all sources real. A gated debug log (env MAL_COUNCIL_DEBUG, off by default) prints the masked raw text on a parse failure, never a key value.
- SEC EDGAR requires a User-Agent header with app name and contact email. No header means blocked requests. Fair-use limit around 10 requests per second.
- ClankApp response shape needs verification against live data before trusting parsers.
- Alpaca paths verified for paper orders and market data. No live brokerage path.
- Record real API responses as test fixtures for every external feed. One malformed parse in a money loop introduces silent bad data.
- Base-check gate runs on Claude Haiku 4.5 through the same Anthropic Messages client as the council secondary (`llm_consensus.providers.anthropic_request`). It reuses ANTHROPIC_API_KEY, so no new credential is needed; without that key the gate degrades to the permissive offline mock. Cost per gate call is well under a cent.

## GUI Plan (dashboard/control-surface overhaul — next major effort)

The dashboard is a first-class control surface, not a read-out. The overhaul (after paper-loop stability is proven) exposes the layered decision system as direct controls, with one inviolable rule: **no control in the UI can weaken the RiskGate.** Layer-1 hard limits are editable only through the typed-`CONFIRM` L1 editor and are still validated at load (a change that would enlarge a loss/exposure ceiling, shrink a quality gate, or disable a safety toggle is rejected); every other control is subordinate.

- **Per-layer enable toggles, safety ALWAYS ON.** Each decision layer (adaptive, LLM council, `dnn_advisory`, whale, `rl_advisory`) gets an on/off toggle. The Layer-1 static-safety layer has NO off switch — it is rendered on and locked. Turning a layer off drops its factor from the ensemble; it can never turn off the gate.
- **Per-model council toggles.** Each council slot (OpenAI / Anthropic / Google) toggles independently, plus the Claude Haiku base-check gate, so the operator can run a cheaper or narrower council without code changes.
- **Champion/challenger controls with MANUAL promote + rollback.** For `dnn_advisory` (and `rl_advisory` when enabled): view champion vs challenger metrics (walk-forward Sharpe, drawdown, provenance, n_samples), a **manual Promote** button gated on `meets_promotion_criteria`, and a **Rollback** to the prior champion. Promotion is never automatic.
- **RL enable toggle, default OFF, behind the 500-fill gate.** A dedicated `rl_enabled` switch that stays off by default and is disabled in the UI until the real closed-fill count reaches `rl_min_real_fills` (default 500); the panel shows current fills / gate. No synthetic-data training path is ever offered.
- **Weight sliders grouped by layer.** Ensemble weights presented grouped by layer (rule-based, LLM, dnn_advisory, whale, rl_advisory), each with lock-against-adaptive and enable/disable, and a live "preview verdict on weight change" before applying.
- **Regime override.** Manual override of the detected regime (trend / range / neutral) for testing and for operator judgement, with a clear "auto" default and a visible banner when overridden.
- **Budget dial.** Operator-adjustable council daily budget + per-symbol cooldown (the cost controls), surfaced as a dial with the current spend against it.
